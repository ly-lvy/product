# 产品功能架构图技能

## 0. 核心原则

本技能用于生成“能直接放进产品汇报、PRD 前言、客户方案或售前材料”的产品功能架构图，而不是规则清单或技术组件堆叠图。

最终图应让读者快速理解：

1. 产品是什么。
2. 支持哪些业务场景。
3. 提供哪些应用输出。
4. 具备哪些核心产品能力。
5. 哪些公共服务或基础能力支撑上层能力。
6. 连接哪些外部数据、设备或系统。
7. 基础版、增强版、可选扩展分别覆盖哪些能力。

内部决策、生成逻辑、图型判断、自检结果、规则解释，严禁出现在最终图中。

---

## 1. 内部思考与生成流程

在输出最终架构前，请在思考区完成以下推导，切勿将推导过程暴露在最终看板、HTML、Mermaid、Markdown 或 PPT 中：

1. 定位分析：明确用户场景，提炼一句话产品定位，形式为“面向...提供...解决...形成...价值”。
2. 架构选型：默认采用“标准功能分层架构”，除非用户明确强调部署、技术拓扑、服务组件，才转为技术架构；明确强调来源、流转、处理、存储、输出，才转为数据流架构。
3. 边界清洗：剔除技术组件堆叠，将 MySQL、InfluxDB、Prometheus、LakeHouse 等技术名转成产品能力名；将第三方系统、设备、传感器、客户数据库等外部依赖划为输入源，不作为产品模块。
4. 模块归类：把功能点整理到业务场景、应用输出、核心能力、公共支撑、多源数据层中。
5. 版本映射：如果存在基础版、增强版、可选扩展，要映射到具体模块或能力分组，不只写在标题或图例里。
6. 视觉生成：按用户要求输出 HTML、Mermaid、Markdown、PPT 或图片就绪内容，并在最终图内隐藏内部推导。

---

## 2. 架构画布绝对禁用

严禁在最终可视化图形中出现以下内容：

- 任何形式的自我辩解与规则解释，例如“本图基于功能视角出发...”。
- 图形选型决策、内部推理、自检通过报告、流程检查清单。
- 除非用户指定，否则禁止出现大面积“角色/岗位”纵向泳道。
- 大面积左侧边栏专门解释产品边界或版本映射。
- 当父级区域已经标明版本时，禁止在每张卡片上重复堆叠微小“基础版/增强版”标签。
- 技术组件作为一级产品模块，例如把 InfluxDB、Prometheus、LakeHouse、BI Server 直接作为主模块标题。
- 红、黄、绿、蓝、紫同时作为大面积主色块。

## 3. 产品定位

内部先写一句产品定位，用于锚定标题、副标题和模块取舍：

> 面向【目标用户/业务场景】，产品提供【核心能力】，解决【核心问题】，最终形成【输出结果/业务价值】。

示例：

> 面向港口、机场、工厂等物流场站，Cactus 提供多源数据接入、边缘处理、运营分析与 AI 增强能力，解决现场数据分散、处理滞后和运营洞察不足的问题，支撑运行监控、效率分析和优化闭环。

最终可在标题或副标题中体现定位，但不要把“定位分析”这类内部字样写进图里。

---

## 4. 产品边界与部署形态

生成前先把内容分成以下几类。

### 4.1 直接产品能力

产品自身提供给用户、管理员、运营人员或外部系统的能力。

示例：

- 数据接入。
- 数据清洗与标准化。
- 指标计算。
- 作业过程分析。
- 异常分析。
- 运营看板。
- 报告生成。
- 告警推送。
- API 输出。
- 权限配置。
- AI 解读。

### 4.2 外部依赖

客户已有或第三方提供的输入对象，不是产品模块。

示例：

- 设备、传感器、车辆、岸桥、场桥、产线设备。
- TOS、FMS、WMS、ERP、MES、TOC、QP 等业务系统。
- 数据库、文件、消息源、第三方 API、对象存储。

外部依赖可以放在“输入源”“外部系统”“多源数据来源”等弱化区域中，不要与产品能力同等强调。

### 4.3 边缘侧能力

部署在客户现场、边缘终端、本地服务器、工控机或本地节点上的能力。

示例：

- 本地数据采集。
- 协议适配。
- 字段映射。
- 边缘实时计算。
- 本地缓存。
- 断点续传。
- 本地运行监控。
- 本地运维控制台。
- 边缘侧数据清洗。

### 4.4 云端 / 后台 / AI 能力

部署在云平台、中心平台、AI 后台或远程服务中的能力。

示例：

- AI 建模。
- 瓶颈识别。
- 趋势预测。
- 自动报告生成。
- 多场站对比。
- 模型训练。
- 远程运维。

### 4.5 可选扩展

专业版、增值版、二期或可选能力。

示例：

- AI 解读。
- 优化建议。
- 方案库复用。
- 多场站对比。
- 控制策略下发。
- 客户门户。

### 4.6 最终输出

用户、系统或管理者能直接看到、收到或调用的结果。

示例：

- 看板。
- 报告。
- 告警。
- API。
- 消息推送。
- AI 解读。
- 优化建议。
- 管理驾驶舱。
- 本地运维控制台。
- 下游系统同步。

### 4.7 反馈闭环

只有当反馈能帮助表达产品价值时才添加闭环。区分：

- 数据反馈：执行结果、运行状态、用户操作回流用于统计、复盘或模型改进。
- 建议反馈：AI 建议被采纳、拒绝或调整后沉淀为经验、口径或方案资产。
- 控制反馈：只有产品真实支持策略或指令下发时才使用。

若产品不支持直接控制，使用“人工确认、运营执行、结果反馈、经验沉淀、模型迭代”，不要写“自动控制”或“闭环控制”。

---

## 5. 图型选择

图型选择只在内部完成，不要渲染到最终画布。

默认使用“产品功能架构图”。仅在用户明确要求时使用其他图型：

- 技术架构图：用户强调部署、数据库、中间件、网关、集群、容器、服务、协议、基础设施。
- 数据流架构图：用户强调数据来源、流转、处理、存储、建模、血缘、输出。
- 页面信息架构图：用户强调菜单、页面、导航、入口、权限、页面结构。
- 业务闭环图：用户强调问题发现、原因分析、建议生成、执行反馈、方案复用、持续改进。
- 客户方案能力图：用户强调客户场景、痛点、能力、输出、价值、交付。

---

## 6. 默认产品功能架构结构

大多数产品功能架构图使用常规分层结构，自上而下组织：

1. 业务场景 / 场景标签。
2. 应用输出层。
3. 核心能力层。
4. 公共支撑层。
5. 多源数据层。
6. 外部依赖与反馈闭环作为轻量辅助区域。

不要把所有产品都强行做成等高表格。结构要清晰、克制、可读。

---

## 7. 分层内容规则

### 7.1 业务场景层

说明产品用于哪里。

使用场景化表达：

- 港口码头作业。
- 机场物流场站。
- 工厂内部物流。
- 仓储 / 园区运营。
- 管理决策汇报。
- 多场站运营管理。

每个场景应能映射到 1-3 个应用输出或核心能力，但映射关系不必占据画面主体。对于边缘或云边协同产品，业务场景通常使用顶部标签呈现，而不是做成大层级。

### 7.2 应用输出层

说明用户直接看到、使用或接收什么。

默认按产品模块组织，不按角色组织。

示例：

- 运营总览。
- 效率分析中心。
- 状态监控中心。
- 告警中心。
- 停时管理。
- 报告中心。
- 管理驾驶舱。
- 开放接口服务。
- 消息推送。
- 本地运维控制台。
- AI 解读报告。

角色信息可作为模块小备注出现，例如“面向现场运维”“面向管理层”，但不要默认单独创建角色视角层。

### 7.3 核心能力层

说明产品实际处理、计算、分析、诊断、预测或推荐什么。

示例：

- 作业过程建模。
- 指标计算。
- 状态识别。
- 效率分析。
- 异常检测。
- 瓶颈定位。
- 趋势预测。
- 优化建议。
- 报告生成。
- 方案沉淀。
- 任务编排。
- 数据建模。
- 空间拓扑与数字孪生建模。
- 场站多源动态调度引擎。
- 设备冲突与轨迹预测。
- 资源统筹编排。

### 7.4 公共支撑层

说明支撑配置、治理、监控、安全和复用的通用能力。

示例：

- 规则引擎。
- 指标口径管理。
- 通用配置。
- 权限管理。
- 安全审计。
- 数据质量校验。
- 运行状态监控。
- 任务调度。
- 日志追踪。
- 参数配置。
- 组织与角色管理。
- 数据目录。
- 方案库。
- 模型管理。

不要把这一层写成纯基础设施层。

### 7.5 多源数据层

说明产品如何接入外部数据并使其可用。

示例：

- 多源数据接入。
- 协议适配。
- 字段映射。
- 接入调度。
- 数据清洗。
- 数据标准化。
- 数据同步。
- 数据补传。
- 断点续传。
- 数据校验。
- 消息订阅。
- 文件导入。
- API 接入。
- 工业协议与 PLC 适配，如 OPC-UA、Modbus。
- 高频定位流数据接入，如 GPS、北斗、UWB。
- 视频流与 AI 边缘感知接入。
- 空间地理数据（GIS）融合。

外部系统和硬件是输入对象，不是产品功能模块。

---

## 8. 边缘 / 云边协同产品处理

边缘、现场终端、软硬一体、云边协同产品的部署形态会影响表达，但不能自动把正常产品架构替换为四区部署图。

默认处理方式：

- 保持常规产品功能架构。
- 用分组、徽标、小备注体现边缘侧能力和云端/AI 能力。
- 业务场景放在顶部标签。
- 外部依赖视觉弱化。
- 应用输出按产品模块组织，不按用户角色列组织。

仅当用户明确要求强调部署、产品边界、云边协同或左到右产品故事时，才使用四区布局：

> 外部输入 -> 边缘侧基础能力 -> 云端 / AI 增强能力 -> 业务输出

不要因为产品包含边缘能力就自动使用四区布局。

---

## 9. 技术术语转换

产品功能架构图中，一级模块标题不得直接使用数据库、中间件、监控工具、基础设施或框架名称。

避免作为主模块标题：

- InfluxDB。
- Prometheus。
- LakeHouse。
- Kafka。
- Redis。
- MySQL。
- PostgreSQL。
- BI Server。
- Web GUI。
- Docker。
- Kubernetes。
- Nginx。

转换为产品能力表达：

| 技术术语 | 产品能力表达 |
| --- | --- |
| InfluxDB | 时序数据管理 |
| Prometheus | 运行状态监控 |
| LakeHouse | 历史数据沉淀与分析 |
| Kafka | 消息流接入与分发 |
| Web GUI | 本地运维控制台 |
| BI Server | 报表与可视化服务 |
| AI Backend | AI 自动化分析服务 |
| Edge Computing | 边缘实时计算服务 |
| MQTT / OPC-UA / PLC | 工业物联协议适配服务 |
| RTSP / RTMP 视频流 | 视频流与边缘感知接入 |
| GIS / Three.js / WebGL | 空间几何拓扑与数字孪生服务 |

技术术语只在必要时放入弱化的“实现参考”说明中。

## 10. 版本能力映射

如果存在版本、套餐、阶段或可选扩展，必须映射到具体模块或能力分组。

不要只在标题、图例或一句说明里解释版本。

### 10.1 优先使用区域 / 分组标签

当能力天然按版本聚合时，使用父级分组标签。

示例：

- 基础能力。
- AI 增强能力。
- 可选扩展能力。

父级区域已经标明版本时，不要在每张卡片上重复相同版本标签。

### 10.2 模块标签只在必要时使用

当同一层级内混排不同版本能力时，才使用模块级标签。

规则：

- HTML 中标签文本必须可读，建议不小于 13px。
- 不使用低对比度微小胶囊标签。
- 同一版本分组内不要每张卡重复相同标签。
- 标签不能压过模块标题。

### 10.3 非产品内容不打版本标签

不要给以下内容强行打版本标签：

- 业务场景。
- 外部系统。
- 第三方硬件。
- 输入数据源。
- 客户角色。
- 客户行业。

---

## 11. 角色处理

产品功能架构图默认不采用角色优先结构。

只有在以下情况才使用角色视角：

- 用户明确要求角色视图。
- 产品核心差异是角色权限或角色工作流。
- 图型是页面信息架构、权限架构、用户旅程或多角色产品矩阵。

否则：

- 应用输出层按产品模块组织。
- 角色信息只作为模块小备注。
- 不创建独立的大面积角色视角层。

---

## 12. 层级副标题规则

层级副标题必须概括该层实际提供的功能能力。

不要写概念解释、规则、设计原则或生成逻辑。

示例：

| 层级 | 不推荐 | 推荐 |
| --- | --- | --- |
| 应用输出 | 用户直接使用或接收 | 输出看板、报告、接口与告警结果 |
| 核心能力 | 产品真正处理和判断的能力 | 完成作业识别、指标计算与异常诊断 |
| 公共支撑 | 配置、治理、监控和复用 | 提供规则、质量、调度与权限支撑 |
| 数据接入 | 外部数据进入并标准化 | 接入多源数据并完成映射清洗 |
| AI 增强 | 面向自动解读、预测分析与运营优化 | 提供瓶颈识别、趋势预测与报告生成 |

---

## 13. 业务闭环

必要时在底部添加一条轻量闭环：

> 采集 -> 处理 -> 计算 -> 洞察 -> 建议 -> 执行 -> 复用

闭环必须保持轻量，不要做成与主架构竞争的大卡片区域。

如果不支持自动控制，使用：

- 人工确认。
- 运营执行。
- 建议采纳。
- 结果反馈。
- 经验沉淀。

不要写“自动控制”或“闭环控制”，除非产品确实支持。

---

## 14. 视觉与美学规范

为保证输出具备可直接用于汇报的专业质感，必须遵循以下规范。

### 14.1 配色克制

- 主色调统一使用科技蓝 / 深青色，例如 `#1E3A8A` 或 `#0D9488`。
- 增强 / AI 能力微量使用群青 / 科技紫，例如 `#6D28D9`。
- 背景统一采用浅灰底色，例如 `#F3F4F6`。
- 外部依赖使用浅灰和弱强调。
- 可选扩展使用低饱和金色或虚线边框。
- 输出模块使用白色卡片与深色边框，不使用大面积红色块。
- 严禁五颜六色，严禁给每个层级分配不同强色。

### 14.2 网格对齐

- 若生成 HTML，必须使用 `display: grid` 或 `display: flex` 布局。
- 各层级内部的模块卡片应等宽或整齐错落。
- 严禁连线交叉交织。
- 产品功能架构默认使用常规分层布局。
- 只有用户明确要求部署或边界故事时，才使用分区或泳道式布局。
- 不在单页内混用分层布局和部署泳道布局。
- 不使用大面积左侧边栏表达边界或版本映射。
- 边界与版本信息应嵌入小备注、分组标题或徽标中。
- 业务场景不是主内容时，使用顶部标签呈现。
- 底部反馈闭环保持为一条轻量流程线。

### 14.3 可读性

- 尽量不超过 5 个架构层级。
- 每层尽量不超过 5 个主模块。
- 单页主模块总数建议不超过 18 个；超过时应分组或拆页。
- 模块标题短而清晰。
- 模块说明简洁。
- 避免密集技术术语。
- 每条箭头必须有明确含义。

---

## 15. 网页输出规则

生成 HTML 时：

- 使用适合汇报的 16:9 画布，通常为 1600 x 900 或 1600 x 1000。
- 隐藏内部决策与规则。
- 不渲染图型判断。
- 不渲染自检内容。
- 除非用户要求，不使用大面积角色视角区域。
- 不使用重复且不可读的版本标签。
- 使用清晰可读的排版；如果使用版本标签，文本建议不小于 13px。
- 层级副标题聚焦功能能力。
- 外部依赖视觉弱化。
- 技术实现说明放在弱化角落；不必要时省略。
- 页面拥挤时减少模块或拆分页面。

---

## 16. 文本响应默认结构

当用户只要求产品架构内容、未要求最终视觉文件时，输出：

1. 产品定位。
2. 架构结构。
3. 产品边界摘要。
4. 分层架构与模块。
5. 版本映射。
6. 反馈闭环。
7. 可视化建议。

不要输出冗长内部 checklist，除非用户明确要求查看 skill 逻辑。

---

## 17. 最终检查

最终交付前确认：

1. 画布中没有内部判断、规则说明、自检结果或流程 checklist。
2. 输出是产品功能架构，而不是技术组件清单。
3. 外部依赖没有被画成产品功能模块。
4. 技术术语已转译为产品能力，必要技术名只在弱化说明中出现。
5. 业务场景、应用输出、核心能力、公共支撑和多源数据层关系清晰。
6. 基础版、增强版、可选扩展的能力映射可读。
7. 没有默认使用角色优先布局。
8. 配色克制，主色统一，AI 色只作少量强调。
9. HTML 使用 grid 或 flex，模块对齐整齐。
10. 底部闭环轻量，不喧宾夺主。
